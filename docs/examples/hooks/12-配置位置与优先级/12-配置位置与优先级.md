# Hook: 配置位置与优先级

> 指南 4.1 + 4.6 — Hook 可在全局、项目、项目私有三个位置配置，所有来源的 Hook 会合并执行

## 目录结构

```
# 全局配置
~/.claude/
├── settings.json                  ← 全局 Hook（所有项目生效）
│
# 项目配置
my-project/
├── .claude/
│   ├── settings.json              ← 项目 Hook（团队共享，提交 VCS）
│   └── settings.local.json        ← 项目私有 Hook（不提交 VCS，个人配置）
├── src/
└── package.json
```

## 说明

```
Hook 配置三个位置:

  1. 全局:     ~/.claude/settings.json
     → 对所有项目生效
     → 适合个人通用 Hook（如审计日志、安全检查）

  2. 项目:     <project>/.claude/settings.json
     → 只对当前项目生效
     → 提交到 VCS，团队共享
     → 适合项目规范（如 lint、格式化）

  3. 项目私有: <project>/.claude/settings.local.json
     → 只对当前项目生效
     → 不提交到 VCS（在 .gitignore 中排除）
     → 适合个人偏好（如通知方式、调试 Hook）

优先级规则:
  → Hooks 是合并（MERGE）的，不是替换的
  → 同一事件的所有来源 Hook 都会执行
  → 执行顺序: 全局 → 项目 → 项目私有
  → 同名 Hook 不会去重（每个都执行）
```

## 使用示例

### 示例 1：全局 Hook + 项目 Hook 同时执行（合并行为）

```
# ~/.claude/settings.json（全局）
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo '全局: 文件修改记录' >> ~/.claude/global-audit.log"
          }
        ]
      }
    ]
  }
}

# my-project/.claude/settings.json（项目）
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix $FILE_PATH"
          }
        ]
      }
    ]
  }
}

→ 用户在 my-project 中编辑文件:

  PreToolUse Hook 执行顺序:
  1. 全局 Hook: echo '全局: 文件修改记录' >> ~/.claude/global-audit.log ✅
  2. 项目 Hook: npx eslint --fix $FILE_PATH ✅
  3. 两个 Hook 都执行了！

→ 用户在 other-project 中编辑文件:
  1. 全局 Hook: 执行 ✅
  2. 项目 Hook: 不存在，跳过

效果:
  → 全局和项目 Hook 合并执行，不会互相覆盖
  → 全局 Hook 在所有项目中生效
  → 项目 Hook 只在特定项目中生效
```

### 示例 2：项目共享 Hook — 团队统一 lint 规范

```
# my-project/.claude/settings.json（提交到 Git）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/auto-lint.sh",
            "async": true,
            "statusMessage": "Auto-linting..."
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/check-branch.sh"
          }
        ]
      }
    ]
  }
}

# .claude/hooks/check-branch.sh
#!/bin/bash
BRANCH=$(git branch --show-current)
if [ "$BRANCH" = "main" ]; then
  echo '{"permissionDecision": "ask", "permissionDecisionReason": "直接修改 main 分支需要确认"}'
fi

→ 团队成员 A 克隆项目:
  → 自动获得 lint + 分支检查 Hook ✅

→ 团队成员 B 克隆项目:
  → 同样获得 lint + 分支检查 Hook ✅

→ 所有人编辑 main 分支都会弹窗确认

效果:
  → 项目 Hook 通过 Git 共享给整个团队
  → 统一的开发规范（lint、分支保护）
  → 新成员加入立即获得相同的 Hook 配置
```

### 示例 3：私有 Hook — 个人调试和通知

```
# my-project/.claude/settings.local.json（不提交 Git）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/my-notify.sh",
            "async": true
          }
        ]
      }
    ],
    "SessionEnd": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/my-timesheet.sh"
          }
        ]
      }
    ]
  }
}

# .claude/hooks/my-notify.sh（个人通知脚本）
#!/bin/bash
# 发送桌面通知（只有我自己能看到）
osascript -e 'display notification "Claude 修改了文件" with title "Claude Code"'

# .gitignore 中排除:
# .claude/settings.local.json
# .claude/hooks/my-*.sh

→ 项目其他成员看不到这些私有 Hook
→ 只影响我自己的开发环境
→ 不干扰团队的公共 Hook 配置

效果:
  → settings.local.json 是个人专属配置
  → 不提交 VCS，不影响他人
  → 适合个人偏好（桌面通知、工时记录、调试日志）
```

### 示例 4：全局 Stop Hook + 项目 Stop Hook 共存

```
# ~/.claude/settings.json（全局）
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/global-summary.sh",
            "async": true
          }
        ]
      }
    ]
  }
}

# my-project/.claude/settings.json（项目）
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/project-cleanup.sh",
            "async": true
          }
        ]
      }
    ]
  }
}

→ Claude 停止工作:

  Stop Hook 执行:
  1. 全局 Hook: global-summary.sh
     → 生成个人工作日报: ~/.claude/daily-reports/2026-04-05.md
  2. 项目 Hook: project-cleanup.sh
     → 清理项目临时文件: rm -f .claude/*.tmp

→ 两个 Hook 都执行，互不干扰

效果:
  → 同一事件（Stop）可以有多个来源的 Hook
  → 全局: 个人日报（跨项目）
  → 项目: 项目清理（项目特定）
  → 合并执行，各司其职
```

### 示例 5：项目私有配置禁用全局 Hook

```
# ~/.claude/settings.json（全局 — 强制安全检查）
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/security-check.sh"
          }
        ]
      }
    ]
  }
}

# my-project/.claude/settings.local.json（项目私有 — 开发时暂时禁用）
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"permissionDecision\": \"allow\"}'"
          }
        ]
      }
    ]
  }
}

→ 编辑文件时:

  PreToolUse 执行顺序:
  1. 全局: security-check.sh → 可能返回 deny
  2. 项目私有: 直接返回 allow

  注意: 由于是合并执行，全局 Hook 先执行:
  → 如果全局返回 deny → 文件被拒绝 ❌
  → 如果全局返回 allow → 继续到项目私有 Hook

  正确的禁用方式: 在项目私有中覆盖全局 Hook:
  → 将全局 Hook 的 security-check.sh 改为空操作
  → 或在全局配置中添加项目白名单

效果:
  → Hooks 是合并的，不能简单"覆盖"或"禁用"
  → 全局 Hook 先执行，项目 Hook 后执行
  → 要禁用全局 Hook 需要修改全局配置或在脚本中做条件判断
  → settings.local.json 适合添加个人 Hook，不适合禁用全局 Hook
```
