# Agent 字段: tools + disallowedTools 组合 + 工具池组装规则

> 指南 3.5「Agent 的工具池」— 完整的工具过滤链

## 目录结构

```
my-project/
├── .claude/
│   └── agents/
│       └── safe-deployer.md         ← ★ 组合使用 tools + disallowedTools
├── scripts/
│   ├── deploy.sh
│   └── health-check.sh
└── package.json
```

## Agent 文件内容

```markdown
# 文件: .claude/agents/safe-deployer.md

---
description: Deploy to staging/production with pre-checks and health verification.
tools:
  - Read
  - Glob
  - Bash
disallowedTools:
  - Edit
  - Write
  - Agent
permissionMode: acceptEdits
maxTurns: 15
---

Deploy the project safely.

## 流程
1. Pre-check: git status, npm test
2. Deploy: bash scripts/deploy.sh {env}
3. Verify: bash scripts/health-check.sh
```

## 说明 — 完整工具池组装（指南 3.5）

```
Agent 启动时，工具池按以下步骤组装:

Step 1: 取完整工具池
  [Read, Write, Edit, Bash, Grep, Glob, Agent, NotebookEdit, AskUserQuestion, ...]

Step 2: 过滤 ALL_AGENT_DISALLOWED_TOOLS（所有 Agent 都禁用的）
  → 移除一些全局禁用工具

Step 3: 如果是自定义 Agent，过滤 CUSTOM_AGENT_DISALLOWED_TOOLS
  → 移除自定义 Agent 禁用的工具

Step 4: 如果是后台 Agent (background: true)，只保留 ASYNC_AGENT_ALLOWED_TOOLS
  → 本例不是后台 Agent，跳过

Step 5: 如果 tools 有白名单，只保留白名单内的
  tools: [Read, Glob, Bash]
  → 只保留 [Read, Glob, Bash]

Step 6: 如果 disallowedTools 有黑名单，移除黑名单内的
  disallowedTools: [Edit, Write, Agent]
  → 从结果中移除（已在 Step 5 过滤掉了）

最终工具池: [Read, Glob, Bash]
```

## 使用示例 1: 部署到 staging 环境

```
用户说: "部署到 staging"
→ 调用 Agent(agentType: "safe-deployer", prompt: "部署到 staging")

Agent 执行:
  1. Bash("git status --porcelain")        ✅ 检查变更
  2. Bash("npm test")                      ✅ 运行测试
  3. Bash("bash scripts/deploy.sh staging") ✅ 部署
  4. Bash("bash scripts/health-check.sh")   ✅ 健康检查

  ❌ 不能 Write/Edit 代码 → 部署不会意外改源码
  ❌ 不能 Agent → 不会嵌套调用其他 Agent
  ✅ permissionMode: acceptEdits → Bash 命令自动执行
  ✅ maxTurns: 15 → 最多 15 轮防止卡死

工具池推导:
  Step 1: [全部工具]
  Step 5: tools 白名单 → [Read, Glob, Bash]
  Step 6: disallowedTools 黑名单 → [Read, Glob, Bash]（Edit/Write/Agent 已不在）
  最终: [Read, Glob, Bash]
```

## 使用示例 3: 部署预检失败 — 工具限制的作用

```
用户说: "部署到 staging"
→ 调用 Agent(agentType: "safe-deployer", prompt: "部署到 staging")

Agent 执行:
  1. Bash("git status --porcelain")   ✅ 发现有 3 个未提交文件
  2. Bash("npm test")                 ❌ 2 个测试失败！

  → Agent 不能做以下操作:
    ❌ Write → 不能修改 package-lock.json 来锁定依赖
    ❌ Edit  → 不能直接修复失败的测试文件
    ❌ Agent → 不能调用另一个"修复 Agent"来修 bug

  → Agent 只能:
    ✅ Read → 读取失败测试的代码
    ✅ Bash → 运行更多诊断命令
    ✅ 输出文本报告 → 告诉用户哪里出了问题

输出:
  ❌ 部署预检失败，已中止部署

  原因 1: 工作区有未提交文件
    - src/utils/format.ts (modified)
    - src/services/user.ts (modified)
    - .env (untracked)

  原因 2: 测试失败
    - user.service.test.ts: "should handle null email" — TypeError
    - format.test.ts: "should format currency" — Expected "$10" got "10$"

  建议: 先修复测试并提交，再重新部署
```

## 使用示例 4: tools 和 disallowedTools 冲突时的优先级

```
场景: 用户想理解两者同时设置时的行为

假设 Agent 配置为:
  tools: [Read, Write, Edit, Bash]
  disallowedTools: [Edit, Bash]

工具池推导:
  Step 1: 完整工具池 [Read, Write, Edit, Bash, Grep, Glob, ...]
  Step 5: tools 白名单过滤 → [Read, Write, Edit, Bash]
  Step 6: disallowedTools 黑名单过滤 → 移除 Edit, Bash
  最终: [Read, Write]

规则: disallowedTools 优先级 > tools
  → 即使 Edit 在 tools 白名单中，只要也在 disallowedTools 中就被移除
  → 这是一个安全机制: 黑名单是最终防线

实际场景:
  用户说: "我想要一个能写文件但不能编辑已有文件的 Agent"
  配置:
    tools: [Read, Write, Edit, Bash]     ← 白名单包含 Edit
    disallowedTools: [Edit]               ← 黑名单排除 Edit
  结果: Agent 只能 Read + Write（创建新文件）+ Bash
  → Edit 被黑名单强制移除，安全需求得到保证
```

## 使用示例 5: 不同角色 Agent 的工具池对比

```
场景: 项目中有 3 个 Agent，用户说不同的话触发不同的 Agent

Agent A — readonly-analyst:
  tools: [Read, Grep, Glob, Bash]
  → 最终工具池: [Read, Grep, Glob, Bash]

Agent B — code-reviewer:
  disallowedTools: [Write, Edit, NotebookEdit]
  → 最终工具池: [Read, Grep, Glob, Bash, Agent, AskUserQuestion, ...]
  → 比 Agent A 多了 Agent、AskUserQuestion 等

Agent C — safe-deployer（本例）:
  tools: [Read, Glob, Bash]
  disallowedTools: [Edit, Write, Agent]
  → 最终工具池: [Read, Glob, Bash]
  → 最精简，只保留部署所需的最小集合

用户说 "分析架构" → 触发 Agent A → 用 Grep/Glob 全面搜索
用户说 "审查代码" → 触发 Agent B → 用 Agent 嵌套调用子审查
用户说 "部署上线" → 触发 Agent C → 只用 Bash 执行部署脚本

安全层级:
  Agent C > Agent A > Agent B
  （工具最少 = 最安全）
```
