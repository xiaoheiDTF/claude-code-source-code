# Agent 文件位置与优先级

> 指南 3.1「文件位置」+ 3.5「工具池」— Agent 的存放位置、加载顺序和优先级

## 目录结构

```
# 三层 Agent 存放位置:

层级 1（最低）    ~/.claude/agents/*.md           全局个人 Agent
层级 2           <project>/.claude/agents/*.md    项目 Agent
层级 3           <subdir>/.claude/agents/*.md     子目录 Agent

# 具体示例:
~/.claude/                           ← 用户 HOME
├── agents/
│   ├── reviewer.md                  ← 全局 Agent（所有项目可用）
│   └── translator.md                ← 全局 Agent
└── ...

my-monorepo/                         ← 项目根
├── .claude/
│   ├── agents/
│   │   ├── reviewer.md              ← 项目 Agent（覆盖全局同名）
│   │   ├── db-admin.md              ← 项目专属 Agent
│   │   └── test-writer.md           ← 项目专属 Agent
│   └── rules/
├── apps/
│   ├── web/
│   │   ├── .claude/
│   │   │   └── agents/
│   │   │       └── component-dev.md ← 子目录 Agent（最高优先级）
│   │   └── src/
│   └── api/
│       └── .claude/
│           └── agents/
│               └── route-dev.md     ← 子目录 Agent
└── ...
```

## 说明 — 同名覆盖规则

```
同名 Agent: 后注册的覆盖先注册的（项目级覆盖全局级）

假设全局和项目都有 reviewer.md:

  ~/.claude/agents/reviewer.md
    description: "General code reviewer"

  .claude/agents/reviewer.md
    description: "Project-specific code reviewer for React + Fastify"

结果: 项目级覆盖全局级
  → 主 Agent 使用项目级的 reviewer（针对 React + Fastify 优化）
  → 全局的 reviewer 不会被加载

不同名: 全部加载，不冲突

  ~/.claude/agents/translator.md     ← 加载
  .claude/agents/db-admin.md         ← 加载
  → 两个 Agent 都可用，互不干扰
```

---

## 使用示例 1: 项目级覆盖全局同名 reviewer

```
用户: "帮我审查代码"

→ 主 Agent 需要委派 reviewer Agent

查找过程:
  1. ~/.claude/agents/reviewer.md        ← 找到全局版本
     description: "General code reviewer"
  2. .claude/agents/reviewer.md          ← 找到项目版本 → 覆盖全局
     description: "Project-specific code reviewer for React + Fastify"
  3. apps/web/.claude/agents/*.md        ← 无 reviewer.md

最终使用: .claude/agents/reviewer.md（项目版本）

Agent 执行:
  → 使用项目版本中定制的 React + Fastify 审查规则
  → 知道项目用 TypeScript strict 模式
  → 知道项目禁止 class component

优先级效果:
  项目级的 reviewer 专为 monorepo 定制，比全局通用版本更精准。
  全局 reviewer 作为"默认"被项目版本覆盖，用户无需删除全局配置。
```

## 使用示例 2: 子目录级 Agent 精准控制工作范围

```
用户: "在 apps/web/ 下帮我开发一个新的用户卡片组件"

→ 主 Agent 委派 Agent

查找过程（当前工作目录: apps/web/src/）:
  1. ~/.claude/agents/*.md               ← 全局: reviewer, translator
  2. .claude/agents/*.md                 ← 项目: reviewer(覆盖全局), db-admin, test-writer
  3. apps/web/.claude/agents/*.md        ← 子目录: component-dev

最终可用 Agent 列表:
  - component-dev（子目录级）← 优先匹配
  - db-admin（项目级）
  - reviewer（项目级，覆盖全局）
  - translator（全局）
  - test-writer（项目级）

→ 选择 component-dev（最匹配当前工作目录）

Agent 执行:
  → 知道 web 子项目用 React 18 + Tailwind
  → 知道组件放在 src/components/ 目录
  → 知道测试框架是 Vitest

优先级效果:
  子目录级 Agent 提供最精准的上下文。
  monorepo 中不同子项目（web vs api）可以有完全不同的开发 Agent。
```

## 使用示例 3: 全局 Agent 跨项目复用

```
用户: "把这个项目的 README 翻译成英文"

→ 主 Agent 委派 translator Agent

查找过程（假设在 ~/projects/api-gateway/ 项目中，无项目级 agents）:
  1. ~/.claude/agents/translator.md      ← 全局: 找到
  2. .claude/agents/                     ← 项目: 目录为空
  3. 无子目录 .claude/

最终使用: ~/.claude/agents/translator.md（全局版本）

Agent 执行:
  → 读取 README.md
  → 翻译为英文
  → 写入 README.en.md

对比在 my-monorepo 中使用:
  查找过程:
    1. ~/.claude/agents/translator.md    ← 全局: 找到
    2. .claude/agents/                   ← 项目: 无 translator.md
  最终使用: 同样是全局 translator

优先级效果:
  全局 Agent 对所有项目生效，一次配置到处使用。
  如果某个项目需要特殊的翻译规则，只需在同名的项目级文件中覆盖。
```

## 使用示例 4: 同名三级覆盖 — 全局 → 项目 → 子目录

```
目录结构:
  ~/.claude/agents/test-writer.md
    description: "Generic test writer using Jest"
    → 工具: Read, Write, Bash
    → 规则: describe/it 结构, AAA 模式

  .claude/agents/test-writer.md
    description: "Test writer for Vitest + React Testing Library"
    → 工具: Read, Write, Bash
    → 规则: vitest 语法, screen.getByRole

  apps/web/.claude/agents/test-writer.md
    description: "Test writer for web app with MSW mocking"
    → 工具: Read, Write, Bash
    → 规则: vitest + MSW handler 模拟 API

用户: "帮我写测试"（当前在 apps/web/src/components/）

查找过程:
  1. ~/.claude/agents/test-writer.md     ← 全局: Jest 版本
  2. .claude/agents/test-writer.md       ← 项目: Vitest 版本 → 覆盖全局
  3. apps/web/.claude/agents/test-writer.md ← 子目录: MSW 版本 → 覆盖项目

最终使用: apps/web/.claude/agents/test-writer.md（子目录版本）

Agent 执行:
  → 使用 MSW 模拟 API 请求
  → 用 Testing Library 写组件测试
  → 知道 web 子项目的 mock 配置路径

优先级效果:
  三层同名 Agent 逐级覆盖，最内层（子目录）的优先级最高。
  允许不同层级逐步细化 Agent 行为:
    全局: 通用测试写法
    项目: 切换测试框架
    子目录: 加入项目特有的 mock 策略
```

## 使用示例 5: 不同名 Agent 全部共存

```
目录结构:
  ~/.claude/agents/
    ├── reviewer.md       ← 通用代码审查
    └── translator.md     ← 翻译工具

  .claude/agents/
    ├── db-admin.md       ← 数据库管理
    └── test-writer.md    ← 测试编写

  apps/api/.claude/agents/
    └── route-dev.md      ← API 路由开发

用户: "我需要审查代码、管理数据库、写测试"

→ 主 Agent 查找所有可用 Agent（当前在 apps/api/ 下）

查找过程:
  1. 全局: reviewer, translator       ← 全部加载
  2. 项目: db-admin, test-writer      ← 全部加载（与全局不重名）
  3. 子目录: route-dev                ← 加载

最终可用 Agent（全部共存，无冲突）:
  - reviewer（全局）
  - translator（全局）
  - db-admin（项目）
  - test-writer（项目）
  - route-dev（子目录 apps/api/）

→ 主 Agent 按需委派:
  "审查代码" → reviewer
  "管理数据库" → db-admin
  "写测试" → test-writer

优先级效果:
  不同名的 Agent 互不干扰，全部可用。
  用户可以在一个对话中连续调用不同层级的 Agent:
    全局 translator 翻译文档
    项目 db-admin 修改表结构
    子目录 route-dev 创建新 API
  每个 Agent 各司其职，层级只是组织方式，不影响共存。
```
